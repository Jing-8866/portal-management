# 系统门户管理 - 开发需求文档

> **数据库**：MySQL 5.7  
> **后端框架**：Spring Boot 2.7.18 + Apache Maven 3.8.1  
> **前端技术**：原生 HTML5 + CSS3 + JavaScript（ES6+），无需 Node/Vue 构建  
> **JDK 版本**：OpenJDK 1.8  
> **Maven 版本**：Apache Maven 3.8.1  

---

## 1. 需求目的

构建一个**统一系统门户管理平台**，实现以下目标：

- 用户登录后进入门户首页，以**卡片形式**展示各子系统入口；
- 点击卡片进入对应子系统，各子系统拥有独立的页面与业务功能；
- **前后端完全分离**：前端为静态页面，后端提供 RESTful API；
- 支持**多子系统权限管控**：用户只能看到并进入已授权的子系统；
- 支持**角色 + 用户**两级权限分配，便于平台管理员与各子系统管理员协作；
- 各子系统使用**独立数据库**，便于扩展与维护。

---

## 2. 开发说明

- 代码完全分离，结构清晰，模块职责单一。
- 前端页面美观、风格统一，采用响应式布局适配 PC 与移动端。
- 可扩展性强：新增子系统只需增加数据库、后端模块、前端 `systems/` 目录及门户卡片配置。
- 门户首页卡片列表上方提供**搜索功能**（按系统名称、描述模糊匹配）。
- 门户首页暂时包含 4 个子系统卡片：用户管理系统、订单管理系统、数据库管理系统、日志分析平台。每个卡片代表一个独立系统，在 `front/systems/` 下分别维护各自的 HTML、JS、CSS。
- 所有子系统前端页面风格保持一致（侧边栏布局、统计卡片、表格、弹窗、Toast 提示等）。
- 数据库 DDL 及初始化数据放在 `backend/portal-main/src/main/resources/DDL.sql`。
- 敏感配置（数据库密码、JWT 密钥）通过 `backend/config/secrets-dev.properties` 外部加载，不提交到版本库。

## 3. 系统说明

### 3.1 门户首页

| 功能 | 说明 |
|------|------|
| 登录校验 | 未登录跳转 `login.html` |
| 子系统卡片 | 调用 `GET /api/subsystems/my`，仅展示当前用户有权限且**已启用**的子系统 |
| 搜索 | 前端本地过滤卡片（名称、描述） |
| 进入子系统 | 点击卡片跳转 `sys.url` 配置的相对路径 |
| 停用系统 | 状态为停用的卡片灰色显示，不可进入 |
| 管理员操作 | 平台管理员 / 子系统管理员可见「新增子系统」及卡片上的编辑、启停、删除按钮 |

**管理员与普通用户卡片差异：**

- **普通用户**：只看到有 `login` 及以上权限且启用的子系统；
- **管理员**（`PLATFORM_ADMIN` / `SUBSYSTEM_ADMIN`）：看到全部子系统（含停用），可进行门户级子系统 CRUD。

### 3.2 用户管理系统（USER_MGMT）

**定位**：管理全平台登录用户、角色及子系统访问权限。

**访问控制：**

- 需拥有用户管理子系统的 **login** 权限方可进入；
- 用户信息的增删改、导入导出、角色/权限分配等写操作，需为 **平台管理员** 或 **子系统管理员**（前端 `isAdmin()` + 后端 `@PreAuthorize` 双重控制）。

**功能模块：**

| 页面 | 路径 | 功能 |
|------|------|------|
| 用户管理 | `index.html` | 用户列表、搜索/状态筛选、前端分页；管理员可新增/编辑/删除/启停/重置密码；Excel 导入导出 |
| 角色管理 | `roles.html` | 角色 CRUD、启停 |
| 用户角色分配 | `role-assign.html` | 为角色批量分配用户（覆盖式） |
| 查询用户角色 | `user-roles.html` | 按用户查询其关联角色 |
| 子系统权限分配 | `perm-menu.html` | 按角色 / 按用户两个 Tab，为指定子系统分配 `login` / `query` / `admin` 权限 |
| 系统权限查看 | `perm-view.html` | 查看用户有效权限（直接分配 + 角色继承合并） |
| 数据权限管理 | `perm-data.html` | **占位页，功能开发中** |

**用户管理补充能力：**

- 个人信息页 `profile.html`：查看/修改姓名、邮箱、手机、密码；展示角色与有效子系统权限；
- 注册接口 `POST /api/auth/register`：创建用户并绑定角色（管理场景使用）；
- 启动时 `PasswordInitializer` 自动将 DDL 中 `INIT:` 前缀的明文密码加密为 BCrypt。

### 3.3 订单管理系统（ORDER_MGMT）

**定位**：订单的查询、创建、状态流转与统计分析。

| 页面 | 路径 | 功能 |
|------|------|------|
| 订单列表 | `index.html` | 统计卡片、关键词/状态筛选、分页列表 |
| 新建订单 | `order-create.html` | 管理员创建订单（客户名、金额、备注）；订单号自动生成 `ORD-yyyyMMdd-XXX` |
| 订单详情 | `order-detail.html` | 查看单笔订单详情 |
| 订单统计 | `order-stats.html` | 月度汇总、状态分布、近 7 日趋势图 |

**订单状态：** `pending`（待处理）→ `processing`（处理中）→ `completed`（已完成）/ `cancelled`（已取消）

**权限：**

- 所有有 login 权限的用户可查询列表、详情、统计；
- 创建订单、变更状态需管理员角色（后端 `@PreAuthorize`）。

### 3.4 数据库管理系统（DB_MGMT）

**定位**：登记并监控 MySQL 数据库实例，支持连接测试与表结构查看。

| 页面 | 路径 | 功能 |
|------|------|------|
| 实例管理 | `index.html` | 实例卡片、搜索、统计；管理员 CRUD、测试连接、单个/全部刷新 |
| 表结构查看 | `tables.html` | 查看实例下表列表、字段结构、DDL 复制 |

**核心逻辑：**

- 实例元数据存储在 `portal_dbmgmt.biz_db_instance`；
- 「刷新」通过 JDBC 实际探测连接状态，并回写在线/离线、表数量；
- 「查看表」对目标实例建立 JDBC 连接，读取 `information_schema`；
- 管理员可新增外部实例（主机、端口、库名、账号密码）。

### 3.5 日志分析平台（LOG_ANALYSIS）

**定位**：集中展示各子系统操作日志，支持多条件检索。

| 页面 | 路径 | 功能 |
|------|------|------|
| 日志查询 | `index.html` | 级别/来源/日期/关键词筛选；动态统计卡片；管理员可删除 |

**日志来源：**

- 后端 `@OperationLog` 注解 + AOP 切面自动写入 `portal_log.sys_operation_log`；
- 记录字段：用户、子系统编码、级别、操作描述、请求方法与 URI、IP、成功/失败、耗时等。

### 3.6 权限模型（核心规则）

#### 3.6.1 预置角色

| 角色编码 | 角色名称 | 说明 |
|----------|----------|------|
| `PLATFORM_ADMIN` | 平台管理员 | 最高权限，可管理所有子系统与用户 |
| `SUBSYSTEM_ADMIN` | 子系统管理员 | 可管理用户及权限分配 |
| `USER` | 普通用户 | 仅使用被分配的权限 |

#### 3.6.2 子系统权限类型

| 类型 | 含义 |
|------|------|
| `login` | 可在门户看到并进入该子系统 |
| `query` | 子系统内查询类操作（与 login 配合使用） |
| `admin` | 子系统内管理类写操作（创建、修改、删除等） |

#### 3.6.3 权限生效规则

1. **直接分配**：`sys_user_subsystem` 为用户直接授予子系统权限；
2. **角色继承**：`sys_role_subsystem` + `sys_user_role`，用户通过角色间接获得权限；
3. **取并集**：两种方式 UNION，任一来源授权即生效；
4. **有效权限视图**：`v_user_effective_permission` 合并上述两种来源，带来源标识（直接分配 / 角色继承）；
5. **门户可见性**：普通用户仅展示有任意权限且 `status=1` 的子系统；
6. **JWT 角色判定**：登录时取用户所有角色中的最高级别（`PLATFORM_ADMIN` > `SUBSYSTEM_ADMIN` > `USER`）；若角色编码仍为 USER 但拥有 `admin` 类型子系统权限，则提升为 `SUBSYSTEM_ADMIN`。

#### 3.6.4 各系统内部权限控制

- **前端**：通过 `isAdmin()` 隐藏/显示写操作按钮；
- **后端**：敏感接口使用 `@PreAuthorize("hasAnyRole('PLATFORM_ADMIN','SUBSYSTEM_ADMIN')")`；
- **认证**：JWT Bearer Token，除 `/api/auth/login`、`/api/auth/register` 外均需认证。

**前端运行方式：**

- 直接打开 `login.html`，或
- 启动本地静态服务（推荐）：`cd front && python -m http.server 3000`
- 后端 API 默认地址：`http://localhost:8080/api`（在 `js/config.js` 中配置）

├── portal-dbmgmt/                      # 数据库管理子系统业务

**模块依赖关系：**

- `portal-main` 聚合启动，依赖 `portal-common`、`portal-order`、`portal-dbmgmt`、`portal-log`；
- 各业务模块仅包含 Controller / Service / Mapper / Model，数据源在 `portal-main` 中统一配置。

**多数据源对应关系：**

| 数据源 | 数据库 | 模块 |
|--------|--------|------|
| mainDataSource | portal_main | portal-main |
| orderDataSource | portal_order | portal-order |
| dbmgmtDataSource | portal_dbmgmt | portal-dbmgmt |
| logDataSource | portal_log | portal-log |

## 7. 部署与运行

### 7.1 数据库初始化

1. 安装 MySQL 5.7；
2. 执行 `backend/portal-main/src/main/resources/DDL.sql`，创建 4 个库及初始数据。

### 7.2 后端启动

1. 复制 `backend/config/secrets-example.properties` 为 `secrets-dev.properties`，填写数据库密码与 JWT 密钥；
2. 在 `backend` 目录执行：
   ```bash
   mvn clean package -DskipTests
   java -jar portal-main/target/portal-main-1.0.0.jar
   ```
3. 默认端口：`8080`，环境切换通过 `spring.profiles.active`（dev / prod）。

### 7.3 前端访问

1. 启动静态服务或直接打开 `front/login.html`；
2. 使用 admin / admin123 登录门户；
3. 从门户卡片进入各子系统。

## 9. 技术要点摘要

- **认证**：Spring Security + JWT（jjwt），无 Session，Token 有效期默认 24 小时；
- **ORM**：MyBatis Plus，多数据源手动配置（排除 Spring Boot 自动数据源）；
- **操作日志**：AOP 切面 `@OperationLog` 异步写入日志库；
- **密码**：BCrypt 加密；DDL 初始密码使用 `INIT:` 前缀，启动时自动迁移；
- **跨域**：CorsConfig 允许前端静态站点跨域访问 API；
- **Excel**：Apache POI 实现用户导入导出（portal-main 模块）。
